# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A quantitative-finance research project (not an application) studying the **lead-lag
relationship between Kalshi prediction-market contracts and Vanguard sector ETFs**,
using the 2024 U.S. Presidential Election and the Fed's Nov/Dec 2024 rate decisions as
the key information events. The research question: do prediction-market price moves
*lead* sector-ETF returns, or vice versa? Headline finding so far: **no clean
single-direction lead.**

There is no build system, test suite, linter, or dependency manifest. Each Python
script is run directly and writes CSV / Markdown / PNG outputs that the next stage reads.

## Running scripts

Scripts assume they are run from their own directory (paths are relative or
`Path(__file__).parent`-based). Dependencies are installed ad-hoc — there is no
`requirements.txt`/`pyproject.toml`. Stack: `pandas`, `numpy`, `statsmodels`, `duckdb`,
`yfinance`, `matplotlib`.

```bash
# Stage 1 — daily-frequency screening (fast, seconds; CSVs are committed)
cd regression/
python regression.py                # -> regression_screen_results.csv (all 594 pairs, ranked by R²)
python build_significant_pairs.py   # -> significant_pairs.csv (48 deduplicated candidate pairs)

# Stage 2 — high-frequency lead-lag pipeline (the real analysis)
cd leadlag/pipeline/
python leadlag_calendar_time.py     # clock-time lag axis
python leadlag_event_time.py        # event-time lag axis
python leadlag_probit.py            # direction-only test
python leadlag_reliability.py       # then: assigns reliability tier per pair
python leadlag_classification.py    # then: groups 48 pairs, tallies who-leads
```

See `leadlag/pipeline/README.md` §8 for the full dependency graph and run order.

## Data — most of it is NOT in the repo

This is critical: the analysis depends on large data that is gitignored or downloaded
separately. A fresh clone **cannot run the high-frequency pipeline** without it.

- **Raw Kalshi dataset** (~36 GB parquet) — downloaded via `make setup`; see
  `prediction-market-analysis/01_kalshi_data_download.md`. Lands in
  `prediction-market-analysis/data/kalshi/{markets,trades}/` (gitignored).
- **ETF high-frequency data** — `leadlag/etf_hf/*_hf.parquet` (~770 MB, not committed).
- **ETF daily returns** — `etf_data/vanguard_sector_etf_total_returns.csv` (committed),
  generated by `etf_data/download_etf_returns.py` via `yfinance`.
- Committed CSVs (`*_kalshi_etf.csv`, `significant_pairs.csv`, etc.) are pipeline
  *outputs* — you can read the analysis without the raw data, but regenerating requires it.

Several report-generation scripts are intentionally gitignored (`build_pdf.py`,
`make_pair_plots.py`, `make_pair_text.py`, `merge/make_*`, etc. — see `.gitignore`).
They are referenced in the pipeline README but live only locally.

## Architecture: the analysis is a linear pipeline

Data flows one direction; each stage's output CSV is the next stage's input.

```
Kalshi raw (36GB) ──find_election_contracts.py──▶ election_contracts.csv (295, keyword screen)
                  ──extract_contract_trades.py──▶ contract_daily_prices.csv (daily OHLC/VWAP)
                                                          │
etf_data/ daily returns ──────────────────────────────────┤
                                                          ▼
                              regression/regression.py ──▶ regression_screen_results.csv
                              (contemporaneous OLS, rank by R²)        │
                                                                       ▼
                              build_significant_pairs.py ──▶ significant_pairs.csv (48 pairs)
                                                                       │
                                          ┌────────────────────────────┘
                                          ▼  leadlag/pipeline/ (high-frequency)
                              calendar + event + probit ──▶ leadlag_*_kalshi_etf.csv
                                          ▼
                              reliability ──▶ classification ──▶ figures/PDF report
```

- **`regression/`** — Stage-1 *screening only*. Daily-frequency contemporaneous
  co-movement (`etf_return(t) ~ prob_change(t)`), ranked by R². Deliberately does **not**
  test lead-lag (that belongs to the high-frequency stage). `build_significant_pairs.py`
  dedups near-identical contracts by "family" (ticker prefix after stripping `KX`) ×
  ETF, keeping the highest-R² representative → 48 pairs.

- **`leadlag/pipeline/`** — the core. Two lag axes (**calendar**-time vs **event**-time)
  share an identical causal construction so their *only* difference is how lags are
  counted; a **probit** runs the same idea as direction-only. Then reliability tiering
  and classification turn 48 noisy per-pair results into a grouped conclusion.

- **`prediction-market-analysis/`** — upstream contract selection (keyword screen →
  manual curation → daily price extraction). Mirrors the public
  `Jon-Becker/prediction-market-analysis` repo plus project-specific scripts.

### `leadlag_common.py` is the single source of truth

**Every convention lives in `leadlag/pipeline/leadlag_common.py`; change it there to
change the whole pipeline.** The point is that calendar and event analyses differ *only*
in their time-axis definition, never in incidental implementation. Conventions enforced
centrally:

- Timezone always `America/New_York`; market hours filtered to 09:30–16:00 ET.
- ETF series → **log returns**; Kalshi `prob = yes_price/100`, differenced **within a
  trading day** (never across overnight).
- Outliers absorbed by **resample + per-bar median** (no σ-thresholds or point deletion).
- **Causal bars only**: right-edge, right-closed (`label="right", closed="right"`) so a
  bar at time `t` uses only data `≤ t` — no look-ahead. This is the recurring discipline;
  any new sampling code must preserve it.
- Bar size and lag order `K` chosen per-contract by trading activity; multiple testing
  corrected with **BH-FDR** (`add_fdr`). Day fixed effects + day-clustered SE + optional
  ADL self-lag control are built into `run_joint_lag_regression`.

- **`merge/`** pools the 19 contracts into 4 sign-aligned "super-signals" by pooling each
  member's own Δprob — **never concatenating price levels** — to recover degrees of freedom.

## Conventions & gotchas

- **Root-level `Lead lag.py` and `Var.py` are legacy daily-frequency scratch scripts**,
  superseded by `leadlag/pipeline/`. Don't extend them as if they were the real analysis.
- Likewise `leadlag/pair_analyses/` and `leadlag/leadlag revised vison.py` are older /
  hand-made artifacts; the auto-generated pipeline pages supersede the hand-made
  `pair_analyses/` (which had a fabricated narrative and is archived locally).
- Kalshi `yes_price` is in **cents (0–100)**, not decimals — `prob = yes_price/100`.
- Pipeline source comments are **bilingual (Chinese + English)**; match that style when
  editing those files. The English half is authoritative if they ever diverge.
- Editable-vs-regenerated outputs: some `.md` files (e.g. `plots/{tag}_analysis.md`,
  `leadlag_classification_summary.md`) are auto-generated but then **hand-edited** — the
  README warns not to re-run the generator after finalizing, as it overwrites manual edits.
- Research disciplines stated in the pipeline README: real-world events cited in analyses
  must be WebSearch-verified with URLs, and sample/window choices must serve the research
  question.
