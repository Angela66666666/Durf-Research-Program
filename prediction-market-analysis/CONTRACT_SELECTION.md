# Prediction Market Contract Selection — Methodology & File Guide

## Research Question

This project studies whether Kalshi prediction market contracts **lead** Vanguard sector ETF returns — i.e., whether price movements in prediction markets contain information that subsequently shows up in stock prices. The 2024 U.S. Presidential Election and Federal Reserve rate decisions are the main events studied.

---

## Contract Selection Pipeline (Step by Step)

### Step 0 — Download Raw Kalshi Data

**How:** Follow `01_kalshi_data_download.md`

The complete Kalshi historical dataset (~36 GB) is downloaded from Cloudflare R2 Storage using `make setup`. It is stored locally in:

```
data/kalshi/markets/   ← contract metadata (ticker, title, status, volume, close_time)
data/kalshi/trades/    ← every individual trade (timestamp, yes_price, count)
```

Files are in **Parquet format** and must be read with Python (pandas or DuckDB).

---

### Step 1 — Keyword Screen → `election_contracts.csv` (295 contracts)

**Script:** `find_election_contracts.py`

This script scans all market metadata parquets using DuckDB and keeps only contracts that:

1. **Match at least one keyword** from a sector-relevant keyword list:

| Target ETF(s) | Keywords used |
|---|---|
| VFH / VNQ / VPU (Rate-sensitive) | federal reserve, rate cut, rate hike, fed, bps, interest rate |
| VDE (Energy) | oil, gas, energy, drill, pipeline, opec, lng |
| VHT (Health Care) | health, pharma, drug, medicare, medicaid, aca, fda |
| VIS / VAW (Industrials / Materials) | tariff, trade, infrastructure, defense, military, steel, china |
| VGT (Information Technology) | antitrust, tech regulation, ai regulation, tiktok |
| All sectors (Election) | president, trump, harris, election, electoral |

2. **Close time between 2024-09-01 and 2024-12-31** — covering the election campaign period and its immediate aftermath.

3. **Volume > 1,000 trades** — to ensure sufficient liquidity for meaningful price signal.

Each contract is also tagged with a `sector_relevance` label based on which keyword group it matched.

**Output:** `election_contracts.csv` — 295 contracts with columns: `ticker`, `title`, `status`, `close_time`, `volume`, `sector_relevance`

---

### Step 2 — Manual Curation → `selected_contracts.csv` (109 contracts)

The 295 keyword-matched contracts were manually reviewed to remove contracts that were too ambiguous, too short-lived, or duplicative. Some additional contracts relevant to the research question were added from other sources.

The file `contracts_comparison.xlsx` (generated by `color_contracts.py`) visualizes this curation:

| Color | Meaning |
|---|---|
| **Yellow** | In both `election_contracts.csv` and `selected_contracts.csv` — kept |
| **Blue** | Only in `election_contracts.csv` — removed during curation |
| **Green** | Only in `selected_contracts.csv` — added from other sources |

**Output:** `selected_contracts.csv` — 109 contracts, same schema as above.

> Note: `extract_contract_trades.py` currently reads from `election_contracts.csv` (all 295), not `selected_contracts.csv`. The regression and lead-lag analysis effectively operates on the broader pool; the `selected_contracts.csv` represents the intended final set and may be used to filter results downstream.

---

### Step 3 — Extract Daily Prices → `contract_daily_prices.csv`

**Script:** `extract_contract_trades.py`

For each contract in `election_contracts.csv`, this script reads all individual trade records from `data/kalshi/trades/*.parquet` and aggregates to daily granularity:

| Column | Description |
|---|---|
| `ticker` | Contract identifier |
| `title` | Contract question text |
| `sector_relevance` | Sector tag from Step 1 |
| `trade_date` | Date (UTC) |
| `open_yes_price` | First trade price of the day (cents, 0–100) |
| `close_yes_price` | Last trade price of the day (cents, 0–100) |
| `vwap_yes_price` | Volume-weighted average yes price |
| `daily_volume` | Total contracts traded |
| `n_trades` | Number of individual trades |

The **probability** used in regression is `close_yes_price / 100`, which gives the market-implied probability that the contract resolves YES (e.g., 0.72 = 72% chance Trump wins).

**Output:** `contract_daily_prices.csv` — daily price observations for all contracts.

## Key Design Decisions

**Why `close_yes_price / 100` as probability?**
The yes price in Kalshi is quoted in cents (0–100). Dividing by 100 gives the risk-neutral market probability. Day-over-day change in this probability is the signal used in regression.

**Why volume > 1,000 as the liquidity filter?**
Thin contracts have wide bid-ask spreads and erratic prices. The 1,000-contract threshold removes noise while keeping all major election and policy contracts.

**Why September–December 2024?**
This window captures the full run-up to the 2024 U.S. Presidential Election (November 5) and the Federal Reserve's November and December meetings — the two main information events this study focuses on.
